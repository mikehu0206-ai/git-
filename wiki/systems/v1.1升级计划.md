## 技术约束说明

### 1. 莫娜职责边界强化
- **现状**：莫娜是 OpenClaw-only agent，没有 Hermes profile
- **实现**：在 `~/.openclaw/agents/mona.prompt` 中追加职责边界约束
- **风险**：低。但需注意不要重复追加（使用 `StrReplaceFile` 或检查是否已存在）

### 2. Agent 自动创建
- **实现思路**：
  1. 创建模板文件 `~/.openclaw/templates/agent.prompt.template`
  2. 脚本接收职责描述 → 填充模板 → 生成 `~/.openclaw/agents/{name}.prompt`
  3. 自动更新 `openclaw.json` 的 `agents.list`
- **风险**：中。需确保 `openclaw.json` 格式正确，更新前自动备份

### 3. 任务看板
- **实现思路**：
  1. 定义任务状态文件规范（如 `~/.openclaw/tasks/{task-id}.json`）
  2. 定时脚本扫描任务目录，生成 `wiki/systems/任务看板.md`
  3. 看板包含：进行中、待执行、已完成、已归档
- **依赖**：需要莫娜或赛特在任务分配时写入状态文件

### 4. 能力评估体系
- **实现思路**：
  1. 从 `~/.openclaw/agents/{agent}/sessions/*.jsonl` 提取 metrics
  2. 从 `security-audit.log` 统计拦截率
  3. 从 Vault 统计知识库引用次数（通过 Obsidian 反向链接）
  4. 生成 `wiki/systems/Agent能力评估报告_YYYYMMDD.md`
- **难点**：需要定义"任务完成质量"的量化标准

### 5. 多模态能力
- **现状**：当前模型 `custom-ark-cn-beijing-volces-com/ark-code-latest` 不支持图片
- **如需实现**：
  1. 评估 Volcano Ark 的视觉模型（如 `doubao-vision-pro`）
  2. 修改 `openclaw.json` 的模型配置
  3. 验证 OpenClaw 是否支持多模态消息格式
- **建议**：列为 v1.2 或更远期的目标，v1.1 期间先做技术预研

## 执行原则

1. **每完成一项升级，必须更新本文件状态，并通知雅典娜归档变更记录**
2. **所有升级必须通过"天命："指令提交给赛特处理**，禁止手动修改配置文件
3. **升级前自动备份**：`openclaw.json`、`agents/*.prompt`、`schema/rules.md`
4. **升级后执行系统巡检**：`openclaw agent --local --agent athena -m "系统巡检"`
